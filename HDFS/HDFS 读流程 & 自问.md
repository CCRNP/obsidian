## 读流程

1. **客户端发起读请求**  
    客户端通过 `FileSystem.open()` 获取 `FSDataInputStream`，向 NameNode 请求打开文件。

2. **NameNode 返回 LocatedBlocks**  
    NameNode 返回 **LocatedBlocks** 对象，包含：
    
    - 文件的所有 **block 列表**（每个 block 有 ID、长度、生成戳）
        
    - 每个 block 对应的 DataNode 列表（按网络拓扑距离排序）
        
    - 文件的长度等元数据
        
3. **客户端选择 DataNode 并建立连接**  
    客户端为每个 block 选择距离最近的 DataNode（优先本地节点，其次同机架）。  
    如果该 DataNode 不可用，则尝试列表中的下一个。  
    客户端与 DataNode 建立 TCP 连接，**并复用连接池中的连接**。
    
4. **流式读取与校验**  
    DataNode 以 **packet** 为单位传输数据。  
    客户端接收 packet 后验证 **checksum**，如果校验失败：
    
    - 向 NameNode **报告坏块**
        
    - 尝试从另一个副本读取
        
    - NameNode 将该副本标记为损坏并启动副本修复
        
5. **顺序读与预取**  
    对于顺序读，客户端会 **预取（prefetch）** 下一个 block 的数据，减少 I/O 等待。  
    对于随机读，客户端可以 **并发读取多个不连续的 block**。
    
6. **对上层应用透明**  
    `FSDataInputStream` 内部管理 block 的切换和定位，应用只需调用 `read()`，就像读取连续文件一样。
    
7. **资源释放**  
    读取完成后，客户端释放资源，**但 TCP 连接可能被放回连接池复用**。

## 自问

**Q：**

1. HDFS写流程中，DataNode挂掉后，NameNode和客户端如何协同处理？
	
2. NameNode内存大小如何估算？假设有1亿个文件块，需要多大内存？
    
3. 小文件对HDFS有哪些危害？列出至少三种解决方案。

**A：**
### 1. HDFS写流程中，DataNode挂掉后，NameNode和客户端如何协同处理？

当写入管道中的DataNode发生故障时，**客户端**和**NameNode**并不会直接通信来恢复，而是由**客户端**主导一个名为**管道恢复**的过程，NameNode在其中扮演的是块副本状态确认的角色。这个过程非常精巧，具体步骤如下：

1.  **故障检测**：客户端通过`DFSOutputStream`向DataNode管道写入数据包（packet）。当某个DataNode（比如DN2）发生故障（如宕机、磁盘满、网络问题）时，它会主动关闭与客户端的TCP连接。客户端在写下一个数据包时，会通过`IOException`或超时机制检测到这个故障。

2.  **客户端暂停与确认幸存者**：客户端立即停止向管道发送数据，并将写管道的数据流标记为待恢复状态。同时，客户端会维护一个**幸存DataNode列表**（例如，原本管道是Client->DN1->DN2->DN3，如果DN2挂了，幸存者就是DN1和DN3）。

3.  **为所有数据块更新Generation Stamp (GS)**：这是最关键的一步。为了标识故障后数据的"新版本"，客户端会通知所有幸存DataNode，为当前正在写入的**数据块**（block）递增其**Generation Stamp**。GS是一个时间戳，用于区分同一个块的"好"副本和"坏"（故障前写入的）副本。新的GS会确保故障后写入的数据不会被老副本覆盖或混淆。[^1]

4.  **重建新管道并恢复数据**：客户端使用幸存DataNode列表重建一个新的写管道。它会从最近一个被确认的数据包之后的位置开始，继续向新管道（Client->DN1->DN3）发送未被确认的数据包。幸存DataNode如果已经收到了这些数据包，会直接忽略并传递给下游；如果没有，则正常写入。

5.  **NameNode的后续协调（异步）**：管道恢复和数据写入完成后，客户端最终会关闭文件流。此时，NameNode才会参与进来。它接收到关闭请求后，会检查该文件的每个块是否达到了最小副本数。如果由于DN故障导致副本数不足（例如原本3副本变成2副本），NameNode会在集群中**自动调度一个新的DataNode**，从现有副本复制数据，以恢复满额副本数。这个过程是异步的，由NameNode的副本监控线程完成。

**特殊情况处理**：
*   **管道建立时失败**：如果是在创建新块并建立管道时失败，客户端会直接向NameNode请求一个新的块ID和一组新的DataNode，从头开始写。
*   **无法替换故障节点**：HDFS提供了配置策略`dfs.client.block.write.replace-datanode-on-failure.policy`，用于控制是否在恢复时引入新DataNode替换故障节点。默认策略`DEFAULT`会根据副本数和写入阶段决定是否替换。

### 2. NameNode内存大小如何估算？假设有1亿个文件块，需要多大内存？

这是一个非常实际的估算问题。首先需要明确，NameNode内存消耗主要来自**文件、目录和块**的元数据对象。

*   **估算公式**：业界公认的经验公式是，**每个元数据对象（文件/目录/块）大约占用 150 字节的内存**。这个值包括了对象自身的开销以及NameNode维护的索引、队列等结构的摊分。
*   **核心关系**：在估算时，不能只看块的数量，还要看文件数量，因为每个文件至少包含一个块（如果是小文件）。但题目假设有1亿个**文件块**，我们需要推导出对应的文件和目录数量。
*   **合理假设**：为了得到总内存，我们需要一个合理的文件与目录比例。假设每个文件只占用1个块（即文件大小不超过块大小），并且平均每个目录包含500个文件。那么：
    *   文件数 = 1亿个
    *   目录数 = 1亿 / 500 = 20万个
*   **总内存估算**：
    *   文件元数据：100,000,000 × 150 字节 = 15,000,000,000 字节 ≈ **14 GB**
    *   块元数据：100,000,000 × 150 字节 = 15,000,000,000 字节 ≈ **14 GB**
    *   目录元数据：200,000 × 150 字节 = 30,000,000 字节 ≈ **0.03 GB**
    *   **总计 ≈ 28 GB**
*   **实际生产经验**：上述计算是理论下限。在实际生产环境中，由于JVM对象开销、各种队列、缓存以及NameNode为高可用（HA）需要维护的共享 edits 日志等，内存消耗通常比理论值高出不少。根据一些云厂商的实践经验，**1亿个文件块（对应约1亿个小文件）实际需要分配给NameNode的Java堆内存大约在 60 GB 左右**。

**结论**：对于1亿个文件块，建议为NameNode配置 **60 GB 或以上**的JVM堆内存，并密切关注GC日志和内存使用情况。

### 3. 小文件对HDFS有哪些危害？列出至少三种解决方案。

#### **危害：**
小文件是指文件大小远小于HDFS块大小（如128MB）的文件。其危害主要体现在三个层面：

1.  **NameNode内存瓶颈**：这是最直接的危害。每个文件、目录和块都需要在NameNode内存中占用约150字节的元数据。如果有1亿个10MB的小文件，它们的数据总容量（1EB）可能DataNode能存下，但NameNode需要消耗数十GB内存来管理元数据，导致内存成为系统扩展的瓶颈。
2.  **数据访问性能下降**：读取大量小文件需要频繁在DataNode间进行网络寻址和磁盘寻道，这违背了HDFS设计为流式读取大文件的初衷，导致读取效率极低。在MapReduce或Spark计算时，每个小文件可能都会启动一个任务（task），导致任务调度和JVM启动的开销远远超过实际数据处理的时间。
3.  **上层应用（如Impala）压力增大**：类似Impala这样的MPP SQL引擎，会缓存HDFS的元数据。大量小文件会导致Impala的Catalog Server内存暴涨，同时使得查询规划变慢，影响整体查询性能。

#### **至少三种解决方案：**

| 解决方案 | 核心原理 | 优点 | 缺点/适用场景 |
| :--- | :--- | :--- | :--- |
| **1. 合并上传：HAR Files (Hadoop Archives)** | 将多个小文件打包成一个 `.har` 文件。它通过在HDFS上创建一个多层文件系统，对外暴露原目录结构，对内减少了NameNode跟踪的文件数量。 | 对用户透明，访问方式几乎不变，能有效减少NameNode内存使用。 | 创建过程需要运行MapReduce作业；读取性能可能比原始文件还差，不适合作为频繁分析的输入，主要用于**冷数据归档**。 |
| **2. 容器文件：SequenceFile / MapFile** | 将小文件的**文件名作为键（key），文件内容作为值（value）**，写入一个大的SequenceFile中。MapFile是带索引的SequenceFile，支持更快的查找。 | 如果文件内容需要被全量处理，这种方式非常高效；SequenceFile支持块压缩（block compression），节省空间。 | 不能像HAR那样列出所有小文件，除非遍历整个文件。适用于**需要一次性批量处理所有小文件**的场景。 |
| **3. 使用专门的NoSQL数据库：HBase** | 将小文件直接存入HBase。HBase在底层将数据存储在带索引的MapFile中，它非常适合处理**随机读/写**和**小文件实时查询**的场景。 | 支持高效的随机访问和更新，能解决小文件带来的NameNode压力。 | 引入额外的系统组件，增加了架构复杂度。适用于**需要频繁随机访问或更新单个小文件**的场景。 |
| **4. 计算框架层面的优化：CombineFileInputFormat** | 在处理输入时，不是为每个小文件分配一个Map任务，而是将多个小文件逻辑上打包成一个`split`，交给一个Map任务处理。 | 无需修改数据存储格式，直接在计算层面减少任务数，降低任务调度开销。 | 只是治标，不能从根本上解决NameNode的内存压力。适用于**存量小文件较多但无法立即合并**的计算任务。 |
